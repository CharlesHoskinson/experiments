# Omega rustdoc style

The omega-commitment workspace ships post-quantum cryptographic primitives, Plonky3 STARK provers and verifiers, and a state-machine harness running on libp2p+openraft+SQLite. Some of these are *load-bearing* in the soundness sense: a missing precondition or an under-documented invariant in `leaf_hash_v2`, `verify`, or the apply pipeline is the kind of thing that ends a chain. Doc comments here are not garnish; they are part of the protocol's specification surface.

This skill codifies what every public item in the workspace must document, what the cryptographic / soundness-bearing items must document on top of that, what the v0.1 limitations get, and how cross-links between crates and design documents work.

## When this skill applies

- You are authoring a new public function, struct, enum, trait, or module in any `omega-commitment/crates/*` crate.
- You are reviewing a PR that touches the public surface of any of those crates.
- You are retrofitting docs onto an existing crate as part of a documentation pass.
- You are about to mark a `tasks.md` checkbox done that produced new public Rust code.

This skill does NOT apply to:

- Private (non-`pub`) items within a single module — those follow `ab-coding-guidelines` only.
- Test code (`#[cfg(test)] mod tests`, `tests/*.rs`) — tests document themselves through their names and assertions.
- Generated code (`build.rs` outputs).

## What every public item gets

A one-line summary, then optional but contractually-conventional sections.

```rust
/// Builds a v1-tagged leaf hash for a sub-tree leaf.
///
/// The preimage is `"omega:v2:leaf" || sub_tree_id || canonical_index_be ||
/// payload_len_be || payload`, hashed with Blake3. The domain tag closes the
/// classic Merkle second-preimage swap; `canonical_index` is bound into the
/// preimage so a verifier reading a published `item_count` can reject any
/// inclusion proof whose `canonical_index >= item_count`.
///
/// # Examples
///
/// ```
/// use omega_commitment_core::hash::leaf_hash_v2;
/// let h = leaf_hash_v2(1, 0, b"alice");
/// assert_eq!(h.len(), 32);
/// ```
///
/// # Soundness
///
/// The output is collision-resistant against any adversary bounded by Blake3's
/// security level (128 bits). Two distinct `(sub_tree_id, canonical_index,
/// payload)` triples produce different hashes; a triple cannot collide with
/// `node_hash_v2`'s output because the domain tags differ.
///
/// # Limitations
///
/// `payload` MUST be ≤ 64 bytes for the v0.1 verifier circuit (one Blake3
/// compression block). Longer payloads compute correctly but cannot be proven
/// in the v0.1 `OmegaMembershipAir`; see [`crate::v0_1`] for the limitation
/// summary.
pub fn leaf_hash_v2(sub_tree_id: u8, canonical_index: u64, payload: &[u8]) -> [u8; 32] {
    // ...
}
```

The required sections:

- **One-line summary** at the top. Imperative voice. Restating the function name as a sentence is forbidden.
- `# Examples` — at least one runnable example unless the item is a marker trait, a sealed type, or pure machine-readable data (e.g. a `pub const`).
- `# Errors` — required on any function returning `Result<_, _>`. Enumerate every error variant the function can produce, each with one sentence on when it fires. If the error type is `Box<dyn Error>` or `anyhow::Error`, that's a bug — the workspace policy bans untyped errors on public surfaces.
- `# Panics` — required on any function that can panic. Most of this codebase forbids panics (`#![forbid(unsafe_code)]`, no `unwrap()`/`expect()` policy), so this section should be rare. When it appears, list the precondition that triggers the panic. If it's panic-free, omit the section — silence implies safe.

The conventionally-required sections (use as applicable):

- **`# Soundness`** — required on every cryptographic or soundness-bearing item. See "Cryptographic / soundness-bearing items" below.
- **`# Limitations`** — required on every item that has a v0.1 / v0.2 boundary. See "v0.1 / v0.2 limitations" below.
- **`# Implementors`** — list of every concrete impl of a public trait, regenerated by tooling.

## Cryptographic / soundness-bearing items get extra

The items in this category are non-negotiable. You cannot ship them without a `# Soundness` block.

| Item | Why it's soundness-bearing |
|---|---|
| `leaf_hash_v2`, `node_hash_v2`, the v2 domain tags | Collision resistance + domain separation |
| `MerkleTree::build_v1`, the duplicate-payload rejection | Ordering and uniqueness invariants |
| `EMPTY_INDEX_SENTINEL`, the padding-leaf logic | Reject padding-leaf forgery |
| `bundle_root_blake3`, `dual_hash`, the SHA3 drift-detection framing | Drift-not-break-hedge contract |
| `prove_collection`, `OmegaMembershipAir::eval`, the LogUp gluing | What the prover commits to |
| `verify`, `VerifyError`, the public-input binding digest | What the verifier accepts |
| `apply_pipeline`, the nullifier-collision check | Replay protection |
| `ClaimPublicInputs`, the wire-format version | Cross-implementation compatibility |
| `MockLedger::open`, the SQLite `WITHOUT ROWID` schema, the actor pattern | Data integrity + concurrency |
| The `pallas_validate::phase1::validate_tx` wrapper | Cardano-tx phase-1 invariants |

A `# Soundness` block answers three questions in plain English:

1. **What does this function preserve?** — the invariant. State it as a precondition implies postcondition.
2. **What attack does it close?** — a concrete adversary model. Name the class (second-preimage, replay, nullifier-collision, malformed CBOR, padding-leaf forgery, wrong-bundle-root substitution, etc.).
3. **What does it NOT preserve?** — the failure mode if a precondition is violated. Be specific. "Returns garbage" is not specific; "the verifier accepts a proof for a different leaf" is.

Example, the actual `verify` function in `omega-claim-verifier`:

```rust
/// Verifies a Plonky3 STARK proof of Merkle inclusion against a published
/// Ω-Commitment.
///
/// # Errors
///
/// Returns:
/// - `VerifyError::CommitmentMismatch` when the proof's bound commitment digest
///   does not equal `commitment`.
/// - `VerifyError::WrongSubTreeRoot` when the public input `per_sub_tree_root`
///   does not equal the commitment's per-sub-tree root for the claimed
///   `sub_tree_id`.
/// - `VerifyError::DepthMismatch` when the public input `tree_depth` does not
///   equal the commitment's recorded depth for that sub-tree.
/// - `VerifyError::InvalidProof` when Plonky3 rejects the proof (any of: trace
///   constraints unsatisfied, FRI failure, lookup imbalance, invalid envelope
///   bytes).
///
/// # Soundness
///
/// `verify(commitment, public_inputs, proof)` returns `Ok(())` iff there exists
/// a witness `(payload, merkle_path, sub_tree_id, leaf_index, depth)` such
/// that:
///
/// 1. `leaf_hash_v2(sub_tree_id, leaf_index, payload) = current_node_0`.
/// 2. For each `i` in `0..depth-1`, `current_node_{i+1} =
///    node_hash_v2(left_i, right_i)` where `(left_i, right_i)` is the swap of
///    `(current_node_i, sibling_i)` keyed by the `i`-th bit of `leaf_index`.
/// 3. `current_node_{depth-1} = commitment.per_sub_tree_root[sub_tree_id]`.
/// 4. `public_inputs.tree_depth == depth` and
///    `public_inputs.per_sub_tree_root == commitment.per_sub_tree_root[
///    public_inputs.sub_tree_id]`.
///
/// The Blake3 compressions inside conditions 1 and 2 are bound to the
/// embedded `OmegaBlake3Air`'s permutation rows via LogUp; an adversary who
/// supplies inconsistent compression inputs causes the LogUp imbalance check
/// to fail in `verify_batch`.
///
/// The proof's `binding_digest` field commits `(commitment, public_inputs)`
/// into the Plonky3 public values. An adversary who rewrites the envelope's
/// `public_inputs` field while leaving the proof bytes intact triggers
/// `InvalidProof` because the recomputed digest no longer matches.
///
/// # Limitations
///
/// v0.1 restricts `payload` to ≤ 64 bytes (one Blake3 compression block); a
/// proof for a longer payload cannot be produced by the v0.1 prover, and the
/// verifier rejects with `InvalidProof` if the trace's payload exceeds that
/// length. Variable-length leaf preimages require the v0.2 `LeafPreimageAir`,
/// which is out of scope for this crate.
///
/// This function is pure: no I/O, no async, no global state, and no panic on
/// adversarial input.
pub fn verify(
    commitment: &OmegaCommitment,
    public_inputs: &ClaimPublicInputs,
    proof: &ProofBytes,
) -> Result<(), VerifyError> {
    // ...
}
```

This is the level of detail every soundness-bearing function gets. It's not optional, and it cannot be auto-generated.

## v0.1 / v0.2 limitations

The codebase has explicit v0.1 boundaries that v0.2 will lift. Every item that touches one gets a `# Limitations` section naming what's not yet sound and what version of the protocol fixes it.

The known v0.1 boundaries:

| Boundary | Item | v0.2 fix |
|---|---|---|
| Leaf preimage ≤ 64 bytes | `OmegaMembershipAir`, `prove_collection`, `verify` | `LeafPreimageAir` for variable-length leaves |
| C6 PQ signature mocked Ed25519 | `verify_signature_in_circuit` (when added) | SLH-DSA gadget |
| C7 PLUME nullifier deterministic Blake3 | `derive_nullifier` | ERC-7524 PLUME construction |
| `--tap-cardano preview` falls back to wall-clock after 5s silence | `omega-toy-consensus` | Document only |
| SHA3 bundle root is drift-detection, not break-hedge | `bundle_root_sha3` | Truly-independent SHA3 sub-tree (v2.0) |

If your function touches any of these, the `# Limitations` block names the boundary explicitly. A reader must not have to grep the design doc to find out the function is partially sound.

## Crate-level docs

Every crate's `src/lib.rs` opens with a module docstring giving:

1. **One-paragraph elevator pitch** — what the crate does, in plain English. Two or three sentences max.
2. **Pointers to design context** — link to the OpenSpec change that authorised the crate, link to the design spec under `cardano-wiki/docs/superpowers/specs/`, and link to the relevant section of `cardano-wiki/wiki/log.md` if the crate has a non-trivial change history.
3. **v0.1 limitations** — the explicit boundaries this crate honours and the v0.2 follow-ups.
4. **Tier of trust** — is this a soundness-bearing crate, a state-handling crate, or a glue crate? Soundness-bearing crates (`omega-commitment-core`, `omega-claim-prover`, `omega-claim-verifier`) carry a stronger doc bar than glue crates (`omega-experiment` CLI).
5. **Conventions specific to this crate** — anything an outside reader would need to know that is not standard Rust idiom.

Example, `omega-claim-verifier/src/lib.rs`:

```rust
//! Plonky3 STARK verifier for Merkle-membership claims against a published
//! Ω-Commitment.
//!
//! # Overview
//!
//! `omega-claim-verifier` consumes proofs produced by `omega-claim-prover` and
//! returns `Ok(())` exactly when the proof attests to a leaf's inclusion under
//! the bound commitment. The verifier surface is pure: no I/O, no async, no
//! global state, no panic on adversarial input.
//!
//! # Design context
//!
//! - OpenSpec change: [`add-proof-experiment-harness`][1].
//! - Spec scenarios: [`Plonky3 verifier for Merkle membership`][2].
//! - PR-2 review record: [`PR-2-REVIEW-v2.md`][3] (post-soundness-fix).
//!
//! [1]: ../../../openspec/changes/add-proof-experiment-harness/
//! [2]: ../../../openspec/changes/add-proof-experiment-harness/specs/proof-harness/spec.md
//! [3]: ../../../openspec/changes/add-proof-experiment-harness/PR-2-REVIEW-v2.md
//!
//! # Tier of trust
//!
//! Soundness-bearing. Every public function carries a `# Soundness` block;
//! see `verify` for the canonical example.
//!
//! # v0.1 limitations
//!
//! - Leaf preimages ≤ 64 bytes (one Blake3 compression block). Longer
//!   preimages require v0.2's `LeafPreimageAir`.
//! - Wire format v2: `ClaimPublicInputs` carries `tree_depth` and
//!   `per_sub_tree_root`. v1 envelopes are unsupported and rejected at
//!   parse time.
#![forbid(unsafe_code)]
#![warn(missing_docs)]
```

## Cross-link conventions

Inside one crate:

```rust
/// See [`leaf_hash_v2`] for the leaf-hash construction.
```

Across crates:

```rust
/// Consumes commitments produced by [`omega_commitment_core::Tree::build_v1`].
```

To a design document, use a footnote-style link:

```rust
/// The dual-hash framing is drift-detection, not a break-hedge; see the
/// 2026-05-03 architectural reframing[^1].
///
/// [^1]: <https://github.com/CharlesHoskinson/experiments/blob/main/cardano-wiki/docs/superpowers/specs/2026-05-03-blake3-migration-design.md>
```

To an OpenSpec change:

```rust
/// Implementation per [`add-proof-experiment-harness`][change]; soundness
/// fix in [`PR #2`][pr2].
///
/// [change]: ../../../openspec/changes/add-proof-experiment-harness/
/// [pr2]: https://github.com/CharlesHoskinson/experiments/pull/2
```

## What NOT to write

- **AI-tells.** "vibrant", "tapestry", "delve", "underscore", "showcase", "pivotal", "leverage", "harness the power of". Run `humanizer` on the prose if in doubt.
- **Tautology.** "The `leaf_hash_v2` function returns the leaf hash v2." Self-evident from the name; cut.
- **Pseudo-code in `# Examples`.** Examples must compile under `cargo doc` (use `--no-run` markers if compilation is needed but execution isn't). Don't write `let x = magic_function();`.
- **Restated function name as a sentence.** "Verifies a verification-related Plonky3 STARK proof verifying that ..." — bad. Cut.
- **Apologetic hedging.** "This function might possibly try to ..." — say what it does.
- **References to the implementation.** "This calls `validate_witnesses` then ..." — readers don't care about the internal call graph; they care about the contract.

## What gets a `#[doc(hidden)]`

Anything that is `pub` for cross-crate visibility but not part of the public API. Internal helpers exposed only because Rust visibility doesn't cleanly express "pub within workspace, private outside". The macro hygiene helpers, the `__private` re-exports, the test-only `pub(crate) fn for_testing(...)`. Mark them `#[doc(hidden)]` so they don't appear in `cargo doc` output.

## Workspace lint configuration

Every public crate carries:

```rust
#![forbid(unsafe_code)]
#![warn(missing_docs)]
#![warn(rustdoc::broken_intra_doc_links)]
#![warn(rustdoc::missing_crate_level_docs)]
```

In the workspace `Cargo.toml`:

```toml
[workspace.lints.rust]
missing_docs = "warn"

[workspace.lints.rustdoc]
broken_intra_doc_links = "warn"
missing_crate_level_docs = "warn"
```

`-D warnings` in CI escalates these to errors. New code that lacks docs fails the gate.

## Doc-comment review checklist

When reviewing a PR or your own work, run through this list:

- Every public item has a one-line summary.
- Functions returning `Result` have `# Errors` blocks listing each variant.
- Functions that can panic have `# Panics` blocks listing each precondition.
- Soundness-bearing items have `# Soundness` blocks naming the invariant, the closed attack, and the failure mode.
- Items at a v0.1 boundary have `# Limitations` blocks naming the boundary.
- Crate-level `lib.rs` has the elevator pitch + design-context links + tier-of-trust + v0.1 limitations.
- Cross-links resolve (`cargo doc` exits 0).
- No AI-tells.
- No tautology.
- Examples compile.
- `#[doc(hidden)]` on internal-pub helpers.

If any line is "no", fix before merging.

## Incremental adoption

The skill is enforced going forward. Existing under-documented items get retrofitted in dedicated documentation passes (one crate at a time), tracked under `tasks.md` group 12 (validation gate). The first such pass is captured in the same commit as this skill.
