## Honest scope statement

This plan has **substantial unknowns** in two specific places:
1. **Pallas-traverse Conway support is partial.** The crate has Babbage / Conway types in `pallas-primitives` but its `pallas-traverse::MultiEraBlock` and adjacent navigators may not cover every LedgerState field we need. Each sub-tree task includes an explicit "if pallas can't handle X, fall back to Koios REST for that sub-tree" branch.
2. **Multi-GB CBOR streaming.** Real LedgerState is ~5–15 GB CBOR. Loading into RAM is infeasible. Pallas's streaming API surface is the discovery work for Task 4.

This plan ships a **working pipeline** end-to-end against real mainnet, not a "production-grade Cardano indexer." It's enough to produce a canonical real-data golden vector and prove the architecture survives contact with reality.

## REVISION 2026-05-03 — split UTxO from stake/governance; UTxO comes from a pallas-network LSQ client

Live testing of the headless `cardano-node 10.7.1` + `cardano-cli 10.16.0.0` (Mithril-restored mainnet at epoch 628, slot 186,222,341) surfaced three architectural facts that override Tasks 2, 4–8 of this plan as originally written. Two earlier "discoveries" in an interim revision were partially wrong; this revision supersedes them.

1. **`cardano-cli query ledger-state` does NOT support `--output-cbor`.** The available flags are `--output-json`, `--output-text`, `--output-yaml`. The CBOR Plan-A path is unavailable through cardano-cli.

2. **The `query ledger-state` JSON output strips the UTxO map but keeps everything else.** A live 2.04 GB dump (`ledger_state_20260502_235649.json`) has `stateBefore.esLState.utxoState.utxo = {}` (length 0), but is otherwise complete:
   - `stateBefore.esLState.delegationState.dstate.accounts` — 1,474,666 stake accounts
   - `stateBefore.esLState.delegationState.dstate.genDelegs` — 7 genesis delegations
   - `stateBefore.esLState.delegationState.pstate.stakePools` — 2,940 stake pools
   - `stateBefore.esLState.delegationState.vstate.dreps` — 1,016 DReps
   - `stateBefore.esLState.delegationState.vstate.committeeState` — full committee state
   - `stateBefore.esLState.utxoState.{deposited, fees, ppups}` — protocol-param state, governance proposals, constitution, committee actions
   - `stateBefore.esLState.utxoState.stake.credentials` — 2,499,064 stake credentials
   - `stateBefore.esSnapshots.{pstakeMark, pstakeSet, pstakeGo}` — three snapshots × 1.32M activeStake + 2.94k stakePoolsSnapShot
   - `stateBefore.esChainAccountState.{reserves, treasury}`
   - `stakeDistrib`, `lastEpoch`, `possibleRewardUpdate`, `blocksBefore`/`blocksCurrent` (block-leader counts)

   This single 2.04 GB JSON file is the **complete input for the stake and governance sub-trees** — no extra `query stake-distribution` / `query gov-state` calls are needed. The earlier interim revision that recommended six separate JSON queries was redundant.

3. **`cardano-cli ... query utxo --whole-utxo` is documented "only appropriate on small testnets"** (verbatim from `cardano-cli conway query utxo --help`) and FAILS on mainnet at offset ~978 MB into the response stream with `DeserialiseFailure "Decoding TxIx: More than 16bits was supplied"`. Root cause: the rigorous decoder `Cardano/Ledger/Address.hs:847` reads pointer-address `TxIx` via `decodeVariableLengthWord16`, but the encoder (`putPtr`, line 348) uses variable-length-Word64. Mainnet's historical record contains pointer-address TxOuts whose `TxIx` exceeds 16 bits. PR `IntersectMBO/cardano-cli#1350` ("Add an srp for a cardano-ledger with the UTxO decoding fix", opened 2026-03-19) carries the hotfix but is unmerged as of 10.16.0.0. Conclusion: `--whole-utxo` is the wrong tool for the mainnet UTxO sub-tree and there is no in-the-box cardano-cli path that works.

### Architectural consequence

**Split the v1.0 ingestion into two independent input streams**:

| Source | Sub-trees produced | Status |
|---|---|---|
| `cardano-cli conway query ledger-state` (single ~2 GB JSON) | stake, governance | ✅ Working today |
| `omega-utxo-snapshot` binary (pallas-network LSQ client → CBOR) | utxo, token-policy, script | ✅ Shipped 2026-05-03 |

The UTxO sub-tree is captured at a chosen tip via the local-state-query miniprotocol — `Acquire → BlockQuery::GetUTxOWhole → Release/Done`. This is the canonical v1.0 path. Two fallback options exist if the LSQ path ever breaks:

1. **(Canonical) Pallas-network N2C LSQ client** — `crates/omega-utxo-snapshot/` (already in workspace). Buffers the `BlockQuery::GetUTxOWhole` response as `AnyCbor`, then writes the raw bytes. Pallas's CBOR decoder doesn't share Haskell's Word16-VLE TxIx bug. No node or cli rebuild.

2. **(Fallback) Build cardano-cli locally with PR #1350 patched in.** Clone `IntersectMBO/cardano-cli@master`, add the cardano-ledger source-repo-package stanza pointing at the hotfix commit, `cabal build cardano-cli`. ~30–60 min Haskell build. Fragile — newer cli might not be wire-compatible with our pinned 10.7.1 node.

3. **(Long-term, v1.1) Replay blocks from `~/cardano/db/db/immutable/` via pallas-traverse.** Walk all 8,619 immutable chunks, fold (txIns spent, txOuts created) into a UTxO map. Slower but most general — same engine produces v1.1's `header` + `tx-index` sub-trees for free, so the chain-follower work doesn't go to waste.

`/home/hoskinson/omega-commitment/scripts/dump_ledger_state.sh`:

```bash
#!/usr/bin/env bash
#
# Dump the full Conway-era LedgerState as JSON via a synced headless
# cardano-node. Multi-GB output. Human-invoked only.
#
# This is the STAKE + GOVERNANCE source. The UTxO sub-tree is captured
# separately by Task 2b's omega-utxo-snapshot binary — the JSON path
# scrubs `utxoState.utxo` to `{}` on mainnet (see the 2026-05-03
# REVISION at the top of the v1.0 plan).
#
# Usage:
#   ./scripts/dump_ledger_state.sh [output-path]
#
# Default output: ~/cardano/snapshots/ledger_state_$(date +%Y%m%d_%H%M%S).json
#
# Required environment:
#   CARDANO_NODE_SOCKET_PATH — path to the headless cardano-node socket

set -euo pipefail

DEFAULT_OUT="$HOME/cardano/snapshots/ledger_state_$(date +%Y%m%d_%H%M%S).json"
OUT_PATH="${1:-$DEFAULT_OUT}"

if [[ -z "${CARDANO_NODE_SOCKET_PATH:-}" ]]; then
  echo "ERROR: CARDANO_NODE_SOCKET_PATH not set." >&2
  echo "       Example: export CARDANO_NODE_SOCKET_PATH=\$HOME/cardano/socket/node.socket" >&2
  exit 1
fi

if [[ ! -S "$CARDANO_NODE_SOCKET_PATH" ]]; then
  echo "ERROR: socket not found at $CARDANO_NODE_SOCKET_PATH" >&2
  echo "       cardano-node must be running and fully synced." >&2
  exit 1
fi

if ! command -v cardano-cli >/dev/null 2>&1; then
  echo "ERROR: cardano-cli not in PATH. See scripts/setup_headless_node.md step 2." >&2
  exit 1
fi

echo "== Checking sync progress =="
TIP_JSON="$(cardano-cli query tip --mainnet)"
SYNC="$(echo "$TIP_JSON" | python3 -c "import json,sys; print(json.load(sys.stdin).get('syncProgress','?'))")"
SLOT="$(echo "$TIP_JSON" | python3 -c "import json,sys; print(json.load(sys.stdin).get('slot','?'))")"
EPOCH="$(echo "$TIP_JSON" | python3 -c "import json,sys; print(json.load(sys.stdin).get('epoch','?'))")"
echo "Tip:  slot=$SLOT epoch=$EPOCH syncProgress=$SYNC"

if [[ "$SYNC" != "100.00" ]]; then
  echo "WARNING: sync incomplete (syncProgress=$SYNC). Dump may be inconsistent." >&2
  read -r -p "Continue anyway? [y/N] " CONFIRM
  if [[ "$CONFIRM" != "y" && "$CONFIRM" != "Y" ]]; then
    exit 1
  fi
fi

mkdir -p "$(dirname "$OUT_PATH")"
echo
echo "== Dumping ledger state (JSON, stake + governance source) =="
echo "Destination: $OUT_PATH"
echo "This produces a ~2 GB JSON file (mainnet at epoch 628, 2026-05)."
echo "cardano-cli 10.16 does not support --output-cbor on this query;"
echo "supported --output-* flags are json|text|yaml only. JSON is the path."
echo

cardano-cli conway query ledger-state \
  --mainnet \
  --out-file "$OUT_PATH"

echo
echo "== Dump complete =="
echo "Path: $OUT_PATH"
echo "Size: $(du -h "$OUT_PATH" | cut -f1)"
echo "Tip slot at dump time: $SLOT"
echo "Tip epoch at dump time: $EPOCH"
echo
echo "Next: feed this JSON into omega-ingest's stake + governance mainnet parsers."
echo "  export OMEGA_LEDGER_STATE_JSON_PATH=$OUT_PATH"
echo "  # (Task 7 / Task 8 consumers — both parse this single JSON file.)"
echo
echo "For the UTxO sub-tree, run Task 2b's omega-utxo-snapshot binary;"
echo "see scripts/setup_headless_node.md step 7b."
```

//! Format auto-detection for ingestion inputs.
//!
//! Synthetic fixtures (v0.9.0) start with a top-level CBOR array (one
//! per sub-tree) that's small (< 1 KB typically). Real Conway
//! LedgerState CBOR dumps from cardano-cli are multi-GB and have a
//! distinctive era-tagged shape.
//!
//! Heuristic: if the file is > 10 MB, assume mainnet. Otherwise,
//! synthetic. Override with the explicit `--format` flag.

use std::path::Path;

#[derive(Copy, Clone, Debug, PartialEq, Eq)]
pub enum InputFormat {
    Synthetic,
    Mainnet,
}

const MAINNET_THRESHOLD_BYTES: u64 = 10 * 1024 * 1024;

pub fn detect_format(path: &Path) -> std::io::Result<InputFormat> {
    let metadata = std::fs::metadata(path)?;
    if metadata.len() > MAINNET_THRESHOLD_BYTES {
        Ok(InputFormat::Mainnet)
    } else {
        Ok(InputFormat::Synthetic)
    }
}

#[cfg(test)]

|---|---|---|---|

## Why Koios is the fallback (not Blockfrost)

- **Free, no API key required** for reasonable use
- **Open-source** (community-run)
- **Comprehensive Conway-era endpoint coverage**
- Rate limits are generous for one-shot ingestion runs

## When Koios is preferred

- pallas-traverse Conway support is incomplete for that sub-tree's CBOR shape
- The sub-tree's CBOR navigation logic is so complex that REST + JSON is faster to implement reliably
- The sub-tree benefits from server-side aggregation (e.g., Koios pre-aggregates first-issuance slot)

- Streaming required (multi-GB CBOR; UTXO set is the canonical case)
- No network dependency desired during the ingestion pass
- Determinism over snapshot-time data — Koios reflects current chain state, which moves; pallas reflects the LedgerState CBOR's frozen state at dump time
